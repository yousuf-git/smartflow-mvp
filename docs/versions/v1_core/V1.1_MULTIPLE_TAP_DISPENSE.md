# SmartFlow MVP — Version 1.1

> Builds on V1. Read [`V1_CORE_DISPENSE.md`](./V1_CORE_DISPENSE.md) first. Target
> entity model for the full system lives in
> [`../TARGET_DB.md/entities_context.md`](../TARGET_DB.md/entities_context.md);
> V1.1 implements the minimal slice aligned with it.

## 1. Goal

V1 dispensed one amount to one tap. V1.1 turns that into a real **order**:

- Plant has **one controller** driving **multiple taps**.
- Single order = up to **2 canes per tap** on up to **2 taps at once** (max 4).
- **Price/litre** and **daily limit** come from the customer's `CustomerType`
  (both are time-series rows; `CustomerType` points at the currently-active row).
- Money is **held** at order time (no wallet debit yet), **debited** when each
  cane acks and starts, **credited back** on stop/fail with undelivered litres.
- Progress for every active cane is visible in one screen.

## 2. Scope

### In scope (V1.1)

- Postgres via async SQLAlchemy.
- Entities (subset of target, columns only as needed):
  `User`, `Customer`, `CustomerType`, `Price`, `Limit`, `Plant`, `Controller`,
  `Tap`, `PurchaseGroup` (V1.1 internal grouping), `Purchase` (one row per cane),

- Seed split into `app/seed/`: `lookups.py` (types/prices/limits),
  `infrastructure.py` (plant/controller/taps), `demo_user.py` (user + opening
  credit).
- Two seeded customer types: **normal** (5/L, 50L/day) and **commercial**
  (4/L, 200L/day). Demo user is **normal**.
- Per-order idle-release timer (90 s) and per-cane retry limit (3 / 60 s).
- Progress WebSocket per order.

### Deferred

- Auth / roles / multiple users — still one hardcoded demo user resolved by email.
- Refunds entity, Staff / Designation, OperatingHours, Lookup / LookupCategory,

- `PurchaseTap` entity from target — V1.1 uses `PurchaseGroup` instead; the two

## 3. Data model (V1.1 subset of target)

```
Price(id, currency, unit_price, timestamp, is_active)        ← time-series
Limit(id, daily_litre_limit, timestamp, is_active)           ← time-series
CustomerType(id, name, price_id→Price, limit_id→Limit)       ← points at current row

User(id, email, first_name, last_name, created_at)
Customer(id, user_id→User, customer_type_id→CustomerType)

Plant(id, name, status, is_active, created_at)
Controller(id, name, plant_id→Plant, status, is_active)
Tap(id, controller_id→Controller, plant_id→Plant,
    gpio_pin_number, status, is_available, label)

PurchaseGroup(id uuid, user_id→User, plant_id→Plant,
              status[active|completed|cancelled], created_at, updated_at)

Purchase(id, group_id→PurchaseGroup, price_id→Price, limit_id→Limit,
         plant_id→Plant, user_id→User, tap_id→Tap, date_time,
         litres_count, litres_delivered, cane_number,
         status[pending|started|completed|partial_completed|failed|cancelled],
         retry_count, retry_window_started_at, reason,
         started_at, completed_at)

WalletTransaction(id, user_id→User, timestamp, amount,
                  transaction_type[credit|debit], purchase_id→Purchase?)
```

**Invariants:**
- ≤ 2 Purchase rows per `(group_id, tap_id)`.
- ≤ 2 taps per `group_id`.
- `balance` = `sum(credit) - sum(debit)` from `wallet_transactions` (ledger).
- `hold` = `sum(litres_count × price.unit_price)` over user's `pending` +
  `started` Purchase rows (live sum; nothing persisted).
- `daily_consumed_litres` = `sum(litres_delivered)` for today's rows.
- `daily_hold_litres` = `sum(litres_count)` for today's `pending`/`started` rows.
- `daily_consumed + daily_hold ≤ limit.daily_litre_limit` (enforced at order time).

## 4. Wallet state machine (ledger-native)

```
order created      → Purchase rows inserted (status=pending); no wallet_tx written
cane ack accepted  → wallet_tx(debit, price × litres_count) written; Purchase.status=started
cane complete      → no wallet movement (full price was debited at start)
cane stopped_early → wallet_tx(credit, unit_price × undelivered_litres); status=partial_completed
cane failed        → same credit for undelivered; status=failed
order cancel /     → pending-only canes flipped to cancelled; no wallet movement
   idle 90 s          (they were never debited)
```

**Retry limit** on START: `retry_count`, `retry_window_started_at` on Purchase.
Window 60 s, max 3 attempts. Over → HTTP 429; user must cancel the cane.

**Auto-STOP on overshoot.** If a `dispensing` progress frame reports
`litres >= litres_count`, the server publishes a STOP command to
`smartflow/cmd/{controller}` and promotes the cane to `complete` (no
refund — full price was debited at START). `litres_delivered` is always
capped at `litres_count` in the DB.

## 5. MQTT

Topics keyed by **controller name** (one controller per plant):

| Topic | Direction | Payload shape |
|---|---|---|
| `smartflow/cmd/{controller_name}` | server→device | `{id, tap_id, action, litres?}` |
| `smartflow/ack/{controller_name}` | device→server | `{id, status, reason?}` |
| `smartflow/progress/{controller_name}` | device→server | `{id, litres, status, reason?}` |

---

### `cmd` payload — server → device

| Field | Type | Constraints / notes |
|---|---|---|
| `id` | integer | `Purchase.id` of the cane being commanded. Must be echoed in `ack` and `progress`. |
| `tap_id` | integer | `Tap.id` the command targets. Must match the Purchase's tap. |
| `action` | `"START"` \| `"STOP"` | `START` → begin dispensing (sent with `litres`); `STOP` → halt early (no `litres`). |
| `litres` | number (optional) | Present only on `START`. > 0 and ≤ `MAX_LITRES` (default 100). Omitted on `STOP`. |

---

### `ack` payload — device → server

| Field | Type | Constraints / notes |
|---|---|---|
| `id` | integer | Must exactly match the `id` from the triggering `cmd`. |
| `status` | `"accepted"` \| `"rejected"` | `accepted` → device will dispense; `rejected` → device refused (tap busy, valve fault, etc.). |
| `reason` | string (optional) | Present only when `rejected`. Human-readable. |

---

### `progress` payload — device → server

| Field | Type | Constraints / notes |
|---|---|---|
| `id` | integer | `Purchase.id` of the dispensing cane. |
| `litres` | number | Cumulative litres delivered. ≥ 0; monotonically increasing; server caps `litres_delivered` in DB at `litres_count` if device overshoots. |
| `status` | `"dispensing"` \| `"complete"` \| `"failed"` \| `"stopped_early"` | `dispensing` → ongoing; `complete` → target reached normally; `failed` → hardware error; `stopped_early` → device halted after receiving `STOP`. Last three are terminal. |
| `reason` | string (optional) | Present on `failed`. Human-readable sensor/hardware description. |

## 6. HTTP API

Under `/api`.

| Method | Path | Description |
|---|---|---|
| `GET` | `/me` | user + customer_type + price/litre + balance + hold + daily-remaining |
| `GET` | `/catalogue` | plants → taps |
| `POST` | `/order` | create order (1–4 canes over ≤2 taps); places hold-by-reference |
| `GET` | `/order/{id}` | read order + its canes |
| `POST` | `/order/{id}/cane/{cid}/start` | publish START; wait for ack; debit; arm idle |
| `POST` | `/order/{id}/cane/{cid}/stop` | publish STOP; refund undelivered; mark `partial_completed` |
| `POST` | `/order/{id}/cancel` | cancel pending canes (no refund needed — never debited) |
| `WS`  | `/ws/order/{id}` | stream per-cane frames |

Error body: `{"detail": {"code": "...", "message": "...", ...}}`.

## 7. Idle auto-release

Timer (`IDLE_RELEASE_SECONDS`, default 90) armed on order create and re-armed
on every successful `/start`. On fire it runs the same logic as `/cancel`
against any `pending` canes. Two outcomes:

- **Order was still entirely pending** (no cane ever started): push
  `cancelled` frames for each cane and close the WS.
- **Some canes already reached a terminal state**: cancel any remaining
  `pending` canes and push frames, but leave the WS open so the client can
  keep inspecting the finished canes.

The timer is explicitly **cancelled** as soon as every cane in the order
reaches a terminal status (through progress, `/stop`, or `/cancel`), so it
never fires on a finished order.

## 7a. Edge cases

Table below is the contract. Every row has an example so you can reproduce
by hand with the AWS IoT MQTT test client against `smartflow/progress/{controller}`.

| # | Situation | Wire trace (example) | Server behaviour | UI behaviour |
|---|---|---|---|---|
| 1 | **Happy path** — hardware delivers exactly `litres_count`. | cane `{id:7, tap_id:2, litres_count:15}` → device sends `{"id":7,"litres":15.0,"status":"complete"}` | Cane → `completed`; no wallet movement (full price was debited at START). Idle timer cancelled if this was the last active cane. | Fill bar 100%, chip = **Complete**, `/me` refreshes (no delta). |
| 2 | **Hardware overshoots while still "dispensing"** — frames report litres ≥ target before the device sends `complete`. | target 15 L; last dispensing frame `{"id":7,"litres":60,"status":"dispensing"}` | Server caps `litres_delivered = 15`, promotes status to `complete`, publishes `{"id":7,"tap_id":2,"action":"STOP"}` to `smartflow/cmd/{controller}`. No refund. | Bar capped at 100%, chip flips to **Complete**, `/me` refreshes. |
| 3 | **Hardware reports `complete` with overshoot** (device ran long but reported complete). | `{"id":7,"litres":17,"status":"complete"}` on a 15 L cane | `litres_delivered` capped at 15. Status = `completed`. No STOP publish (device already stopped itself). | Same as row 1. |
| 4 | **User-initiated STOP mid-dispense.** | `POST /order/{oid}/cane/7/stop` while `litres_delivered=6` of 15 | Server publishes STOP, writes `credit` wallet_tx for `(15 − 6) × unit_price`, sets status = `partial_completed`, pushes frame. | Bar stays at 40%, chip = **Stopped**, balance increases by the refund on next `/me`. |
| 5 | **Device reports failure mid-dispense.** | `{"id":7,"litres":3,"status":"failed","reason":"flow_sensor_timeout"}` | Credit wallet_tx for `(15 − 3) × unit_price`, status = `failed`, reason persisted. | Bar at 20%, chip = **Failed**, refund reflected in header. |
| 6 | **Ack timeout on START.** | `POST /start`; no `smartflow/ack/{controller}` within `ACK_TIMEOUT_SECONDS` | HTTP 504 `{code:"ack_timeout"}`; `retry_count` bumped. No wallet movement. | Toast error; cane stays `pending`, attempt counter shown. |
| 7 | **Retry limit hit.** | 4th `/start` within 60 s | HTTP 429 `{code:"retry_limit"}`; cane still `pending`. User must `/cancel`. | Toast warning; Start button stays enabled but next click re-hits 429 until window rolls. |
| 8 | **Tap busy on START.** | Cane B targets tap 2; cane A on tap 2 is already `started`. | HTTP 409 `{code:"tap_busy"}`. | Start button is disabled in the UI (`canStart=false`) — server check is belt-and-braces. |
| 9 | **Progress frame for an unknown / terminal cane.** | `{"id":999,...}` or repeat of old id | Logged and dropped (no state change). | No UI effect. |
| 10 | **Idle timeout on a never-started order.** | Create order, wait 90 s. | All canes flipped `cancelled` (reason `idle_timeout`). Group status = `cancelled`. No wallet movement (never debited). WS gets `cancelled` frames then `__close__`. | Cards flip to **Cancelled**, countdown chip disappears, WS closes cleanly. |
| 11 | **Idle timeout on a partially-finished order.** | Two canes; one completed, the other left `pending` 90 s. | The still-pending cane is cancelled; WS stays open so the user can see both outcomes. Idle timer is not re-armed. | Cards show mix of **Complete** and **Cancelled**; Done button enables. |
| 12 | **Order cancelled by the user with one cane already started.** | `POST /cancel` while cane 7 is `started`, cane 8 is `pending`. | Cane 8 → `cancelled`; cane 7 is untouched (cannot cancel a started cane — it must STOP). Group status stays `active` until cane 7 terminates. | Cane 8 flips to **Cancelled**; cane 7 continues; cancel button grays out if nothing else pending. |
| 13 | **Insufficient balance or over daily limit at order-create.** | `POST /order` where `litres × price > (balance − hold)` or litres pushes `daily_consumed + daily_hold` over the limit. | HTTP 402 `{code:"insufficient_balance"}` or `{code:"over_daily_limit"}` with numeric details. No rows written. | Error toast with the server message; home screen re-renders with the existing draft intact. |
| 14 | **WS dropped mid-order (network blip).** | Browser loses connection. | Server keeps `GroupRuntime` alive. Future frames queue up. Client closing cleanly is `wasClean=true`; a drop is not. | Toast "Lost live updates." Current impl does not auto-reconnect; user reloads or waits for end-of-order (`/order/{id}` can be polled). |

Invariants worth repeating:

- `litres_delivered` in the DB is never greater than `litres_count`.
- Once a cane is terminal, further progress frames for that cane are dropped.
- Refunds are only written for `stopped_early` and `failed`. `complete`
  never credits, even if `litres_delivered < litres_count` (device reporting
  issue, not a money issue — full price paid at START).
- The WS is closed by the server only in the "order was idle from start"
  case or on shutdown. All other closes are client-initiated.

## 11. Definition of done

- DB boots; `prices`/`limits`/`customer_types` seeded; demo plant/controller/taps
  seeded; demo user + customer + opening credit seeded. Re-boot is idempotent.
- `/api/me` returns balance = opening credit, hold = 0, daily_remaining equal to
  the customer_type's Limit on first boot.
- Creating an order with 3 canes across 2 taps bumps `hold` by `litres × price`.
- Starting each cane writes one `debit` wallet_transaction.
- Stopping mid-dispense writes a `credit` wallet_transaction for undelivered
  litres.
- Cancelling an unstarted order or letting it idle 90 s flips the pending canes
  to `cancelled` without any wallet_transaction row.
- Retry limit (3 / 60 s) returns HTTP 429; only cancel clears the cane.
- Progress WS pushes frames for every active cane concurrently.
