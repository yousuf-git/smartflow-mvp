# V1.2 ‚Äî QR-Based Plant and Tap Selection

## Overview

Replaces the default plant/tap pre-selection on the purchase screen with a QR scan entry point. Users scan a QR code at their dispenser tap to enter the fill flow. A bypass button and print utility support development/testing.

The app always returns to `qr_scan` after an order completes, is cancelled, or the idle timer expires. Suitable for kiosk-style deployment where each customer starts fresh.

## QR Code Format

QR encodes a JSON string:

```json
{"plant_id": 1, "tap_id": 2}
```

| Field | Type | Constraints / notes |
|---|---|---|
| `plant_id` | integer | Must match a plant `id` in the local catalogue; missing ‚Üí error toast, user stays on QR screen |
| `tap_id` | integer | Must match a tap `id` under that plant in the catalogue; missing ‚Üí error toast, user stays on QR screen |

Validation is client-side only ‚Äî the catalogue is loaded once at startup and held in `App.tsx` state.

## QR Scan Screen (`QRScanScreen.tsx`)

### Elements

| Element | Behaviour |
|---|---|
| **Scan QR Code** (primary button) | Opens camera dialog; `CameraScanner` starts scanning |
| **Use defaults** (outlined button) | Skips scan; uses `catalogue.plants[0]` and `plants[0].taps[0]` |
| **Print test QR** (text button) | Generates QR for default plant/tap, opens print dialog |

### Camera dialog

- `CameraScanner` sub-component mounted only while dialog is open (`scanOpen === true`)
- Uses native `getUserMedia({ video: { facingMode: "environment" } })` ‚Äî back camera on mobile, first available on desktop
- `requestAnimationFrame` tick loop draws `<video>` frame to `<canvas>` ‚Üí `jsQR(imageData.data, w, h)` decodes
- On successful decode ‚Üí dialog closes ‚Üí JSON parsed ‚Üí validated against catalogue ‚Üí `enterHome(plant, tap)`
- On camera error or validation failure ‚Üí dialog closes ‚Üí **toast shown** (not inline error)
- Cleanup on unmount: `cancelAnimationFrame` + `stream.getTracks().forEach(t => t.stop())` ‚Äî camera light goes off immediately

## Entry to `home` (both paths)

After QR scan or bypass, `enterHome(plant, tap)` is called:

1. `selectedPlant` set to the resolved plant
2. Draft pre-populated with one cane on the scanned tap (`litres: "10"`)
3. Screen transitions to `home`

`CaneBuilder` receives the selected `plant` and the initially scanned `tap`. Only taps already present in the draft are shown ‚Äî the scanned tap appears first. Additional taps are added via an inline QR scan (see below), not via a dropdown.

## CaneBuilder ‚Äî tap selection changes

`CaneBuilder` was updated in V1.2 to gate tap visibility and add a second-tap scan flow:

| Feature | Behaviour |
|---|---|
| **Active-tap list** | Only taps already in draft are shown; starts with one tap (from QR or bypass) |
| **Add another tap** button | Visible when < 2 unique taps in draft; opens inline `CameraScanner` dialog |
| **Same-tap scan** | Warns via toast: "Tap already selected ‚Äî scan a different tap or add canes here." No cane added |
| **2-tap limit** | "Add another tap" button hidden once 2 unique taps are in draft |
| **Wrong plant scan** | Error toast; no tap or cane added |

## Print Test QR

- Generates QR for `catalogue.plants[0]` / `plants[0].taps[0]`
- Uses `qrcode` npm package (`QRCode.toDataURL`) ‚Äî browser canvas, no server needed
- Opens `window.open` popup with styled HTML; `window.print()` fires on load
- "Save as PDF" from the print dialog produces a usable test QR

## Toast notifications (`Toast.tsx` + `useToast.ts`)

Reusable toast system introduced in V1.2:

- `Toast.tsx` ‚Äî `Snackbar` + `Alert` + `Slide` (up) transition; auto-hides after 4 s; has ‚úï button
- `useToast.ts` ‚Äî hook returning `{ toastProps, showToast(message, severity) }`; spread `toastProps` into `<Toast />`
- Used by both `QRScanScreen` (camera/validation errors) and `CaneBuilder` (same-tap warning, scan errors)
